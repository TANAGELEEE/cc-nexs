# 01.demo-feature — 业务需求

> 演示用例：给后端服务添加 /api/health 健康检查接口。

## 业务诉求

线上需要一个轻量的健康检查接口供负载均衡器探活。当前没有，每次发版都要靠日志判断进程是否真的启动成功。

## 关键场景

1. 负载均衡器每秒调一次 `GET /api/health`，期待 200 + JSON `{"status":"ok"}`
2. 内部监控系统调 `GET /api/health?detail=true` 期待返回 DB / Redis / Kafka 三项依赖的连通状态
3. 启动期间（依赖未就绪）应返回 503，进入就绪后返回 200

## 验收要点（业务侧）

- 健康接口必须 200ms 内返回（不阻塞业务请求）
- detail=true 模式探活失败的依赖项要明确指出
- 接口本身不能成为新的故障点（任何依赖故障不能让健康接口本身崩溃）

## 约束 / 依赖

- 不依赖任何新组件，只查现有 DataSource / RedisTemplate / KafkaTemplate
- 暴露在 `/api/` 前缀下（无需鉴权）
- 与现有 Spring Actuator 不冲突（如果有）

## 不在范围内

- 监控告警系统接入（运维侧自己做）
- 健康指标历史持久化
